What is Git?

5 min read Beginner

What is Version Control?

Imagine you're writing a document and you want to keep track of all the changes you've made over time. You might save copies like document_v1.docx, document_v2.docx, document_final.docx, document_final_FINAL.docx... Sound familiar?

Version control is a system that records changes to files over time so you can recall specific versions later. Instead of creating dozens of copies, version control automatically tracks what changed, when it changed, and who changed it.

So What is Git?

Git is the most popular version control system in the world. Created by Linus Torvalds (the creator of Linux) in 2005, Git helps you:

  • Track changes - See exactly what changed in each file and when
  • Undo mistakes - Easily revert to a previous version if something goes wrong
  • Collaborate - Work with others on the same project without overwriting each other's work
  • Branch out - Experiment with new features without affecting the main project

Why Should You Learn Git?

Git is an essential skill for software developers, but it's useful for anyone who works with files that change over time:

  • Writers tracking document revisions
  • Designers managing project iterations
  • Students collaborating on group projects
  • Anyone who's ever lost work to an accidental save
Git vs GitHub

Git is the version control tool. GitHub is a website that hosts Git repositories online and adds collaboration features. Think of Git as the engine and GitHub as the garage where you park your car.

How Git Works (The Simple Version)

Git tracks your project in a special folder called a repository (or "repo"). Inside this repo, Git keeps a complete history of every change you've saved.

The basic workflow is:

  1. Modify - Change files in your project
  2. Stage - Choose which changes you want to save
  3. Commit - Save a snapshot of your staged changes with a description

Each commit is like a checkpoint in a video game—you can always go back to it if needed.

Common Mistake

Don't skip learning Git because it seems complicated. The basics are simple, and even knowing just 5-6 commands will cover 90% of what you need day-to-day.

Lessons / Installing Git

Installing Git

8 min read Beginner

Check if Git is Already Installed

Before installing, let's check if Git is already on your computer. Open your terminal (or Command Prompt on Windows) and type:

Terminal 
git --version

If Git is installed, you'll see something like:

Output
git version 2.42.0

If you see a version number, you're all set! Skip to the configuration section below.

Installing on Windows

  1. Go to git-scm.com/download/win
  2. Download the installer and run it
  3. Accept the default options (they're fine for beginners)
  4. Click through the installation wizard
Git Bash

The Windows installer includes "Git Bash," a terminal that gives you a Linux-like command line experience. It's the recommended way to use Git on Windows.

Installing on macOS

The easiest way is using Homebrew:

Terminal 
brew install git

If you don't have Homebrew, you can install Git by installing Xcode Command Line Tools:

Terminal 
xcode-select --install

Installing on Linux

Use your distribution's package manager:

Debian/Ubuntu 
sudo apt-get update
sudo apt-get install git
Fedora 
sudo dnf install git

Configuring Git

After installation, you need to tell Git who you are. This information will be attached to every commit you make.

Terminal 
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

Replace "Your Name" and the email with your actual information.

Verify Your Configuration

Terminal 
git config --list
Output
user.name=Your Name user.email=your.email@example.com
Common Mistake

Don't forget to configure your email! If you skip this step, your commits won't be properly attributed to you, which matters when collaborating with others.

Lessons / git init

git init

5 min read

What Does git init Do?

The git init command creates a new Git repository. It's the first command you run when starting a new project that you want to track with Git.

When you run git init, Git creates a hidden folder called .git in your project directory. This folder contains all the information Git needs to track your project's history.

How to Use It

First, create a folder for your project and navigate into it:

Terminal 
mkdir my-project
cd my-project

Then initialize Git:

Terminal 
git init
Output
Initialized empty Git repository in /Users/you/my-project/.git/

What Just Happened?

Git created a .git folder in your project. You can verify this:

Terminal 
ls -la
Output
total 0 drwxr-xr-x 3 you staff 96 Jan 1 12:00 . drwxr-xr-x 5 you staff 160 Jan 1 12:00 .. drwxr-xr-x 10 you staff 320 Jan 1 12:00 .git
Don't Delete .git!

The .git folder contains your entire project history. If you delete it, you'll lose all your commits and version control for that project.

Common Mistakes

1. Running git init in the Wrong Folder

Make sure you're in your project folder before running git init. Running it in your home directory or a parent folder will track way more files than you intend.

2. Initializing Inside Another Git Repository

Don't create a Git repository inside another Git repository (nested repos). This creates confusion and problems. If you're unsure whether you're already in a repo:

Terminal 
git status

If you see "fatal: not a git repository," you're safe to run git init.

Lessons / git status

git status

5 min read

Your Best Friend in Git

The git status command shows you the current state of your repository. It tells you:

  • Which branch you're on
  • Which files have been modified
  • Which files are staged (ready to be committed)
  • Which files are untracked (new files Git doesn't know about)

Using git status

Simply run:

Terminal 
git status

In a Clean Repository

Output
On branch main nothing to commit, working tree clean

With Untracked Files

Let's create a new file and check the status:

Terminal 
echo "Hello World" > hello.txt
git status
Output
On branch main No commits yet Untracked files: (use "git add <file>..." to include in what will be committed) hello.txt nothing added to commit but untracked files present (use "git add" to track)

With Staged Files

After running git add hello.txt:

Output
On branch main No commits yet Changes to be committed: (use "git rm --cached <file>..." to unstage) new file: hello.txt

Understanding the Output

Git status shows files in different categories:

  • Untracked files (red) - New files Git doesn't know about yet
  • Changes to be committed (green) - Files staged and ready for commit
  • Changes not staged (red) - Modified files not yet staged
Pro Tip

Run git status frequently! It's the best way to understand what's happening in your repository. When in doubt, check the status.

Short Status

For a more compact view, use the -s flag:

Terminal 
git status -s
Output
A hello.txt ?? newfile.txt M modified.txt

The codes mean:

  • A - Added (staged)
  • ?? - Untracked
  • M - Modified

Lessons / git add

git add

6 min read

The Staging Area

Before you can save (commit) your changes, you need to tell Git which changes to include. This is called staging, and the git add command is how you do it.

Think of it like packing a box before shipping. You choose what goes in the box (staging), then seal and ship it (commit).

Adding a Single File

Terminal 
git add filename.txt

Adding Multiple Files

Terminal 
git add file1.txt file2.txt file3.txt

Adding All Files

To stage all changed and new files at once:

Terminal 
git add .

The dot (.) means "everything in the current directory and subdirectories."

Be Careful with git add .

Using git add . stages everything, including files you might not want to commit (like passwords, API keys, or log files). Always run git status after to check what you staged.

Example Workflow

Terminal 
# Create some files
echo "# My Project" > README.md
echo "console.log('hello');" > app.js

# Check status - files are untracked
git status

# Stage the files
git add README.md app.js

# Check status again - files are now staged
git status

Unstaging Files

Made a mistake? Remove a file from the staging area:

Terminal 
git restore --staged filename.txt

Or for older Git versions:

Terminal 
git reset HEAD filename.txt

Common Mistakes

1. Forgetting to Stage Before Committing

If you modify a file but don't run git add, the changes won't be included in your commit. Always check git status before committing.

2. Staging Sensitive Files

Use a .gitignore file to prevent sensitive files from being staged:

.gitignore 
# Ignore these files
.env
passwords.txt
node_modules/
*.log

Lessons / git commit

git commit

7 min read

Saving Your Work

The git commit command saves your staged changes as a permanent snapshot in your repository's history. Each commit is like a checkpoint you can return to later.

Basic Usage

Terminal 
git commit -m "Your commit message here"

The -m flag lets you write the commit message inline.

Output
[main 3a2f1b4] Your commit message here 2 files changed, 15 insertions(+) create mode 100644 README.md create mode 100644 app.js

Writing Good Commit Messages

Your commit messages should explain what changed and why. Good messages make it easier to understand your project's history.

Good Examples ✓

  • "Add user login functionality"
  • "Fix navigation menu alignment on mobile"
  • "Update README with installation instructions"
  • "Remove deprecated API endpoints"

Bad Examples ✗

  • "fix" - Fix what?
  • "updates" - What updates?
  • "asdfasdf" - Meaningless
  • "WIP" - Doesn't explain anything
Commit Message Convention

Start with a verb in present tense: "Add", "Fix", "Update", "Remove", "Refactor". Keep the first line under 50 characters.

The Complete Workflow

Terminal 
# 1. Make changes to your files
echo "New content" >> README.md

# 2. Check what changed
git status

# 3. Stage your changes
git add README.md

# 4. Commit with a message
git commit -m "Add new content to README"

Commit Without -m

If you run git commit without -m, Git opens a text editor for you to write a longer message:

Terminal 
git commit

This is useful for writing detailed multi-line messages.

Amending the Last Commit

Made a typo in your commit message? Forgot to include a file?

Terminal 
# Fix the message
git commit --amend -m "New corrected message"

# Or add forgotten files to the last commit
git add forgotten-file.txt
git commit --amend --no-edit
Warning

Only amend commits that haven't been pushed to a shared repository. Amending changes the commit's ID, which can cause problems for collaborators.

Lessons / git log

git log

5 min read

Viewing Your History

The git log command shows you the history of commits in your repository. It's like a timeline of all the changes that have been made.

Basic Usage

Terminal 
git log
Output
commit 3a2f1b4c8d9e0f1a2b3c4d5e6f7a8b9c0d1e2f3a (HEAD -> main) Author: Your Name <you@example.com> Date: Mon Jan 1 12:00:00 2024 -0500 Add user authentication feature commit 1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c Author: Your Name <you@example.com> Date: Sun Dec 31 10:30:00 2023 -0500 Initial commit

Each entry shows:

  • Commit hash - A unique identifier for the commit
  • Author - Who made the commit
  • Date - When the commit was made
  • Message - The commit description

Useful Options

One Line Per Commit

Terminal 
git log --oneline
Output
3a2f1b4 Add user authentication feature 1b2c3d4 Initial commit

Show Last N Commits

Terminal 
git log -5

Show Graph View

Terminal 
git log --oneline --graph --all
Output
* 3a2f1b4 (HEAD -> main) Add user authentication | * 2b3c4d5 (feature) Work in progress |/ * 1b2c3d4 Initial commit

Search Commits

Terminal 
# Find commits with "bug" in the message
git log --grep="bug"

# Find commits by author
git log --author="Your Name"
Pro Tip

Press q to exit the log view when it's showing in your terminal.

Lessons / Branches

Branches

8 min read

What Are Branches?

Branches let you work on different versions of your project at the same time. Think of them like parallel universes for your code.

The main branch (often called main or master) is your primary timeline. You can create new branches to work on features, fix bugs, or experiment—without affecting the main code.

Why Use Branches?

  • Safe experimentation - Try new ideas without breaking working code
  • Parallel development - Multiple features can be developed simultaneously
  • Code review - Changes can be reviewed before merging into main
  • Bug fixes - Quickly create a branch to fix a bug in production

Creating a Branch

Terminal 
git branch feature-login

Switching Branches

Terminal 
git checkout feature-login

# Or use the newer command
git switch feature-login

Create and Switch in One Command

Terminal 
git checkout -b feature-login

# Or with switch
git switch -c feature-login

Listing Branches

Terminal 
git branch
Output
* feature-login main

The asterisk (*) shows which branch you're currently on.

Merging Branches

When your feature is complete, merge it back into main:

Terminal 
# First, switch to main
git checkout main

# Then merge your feature branch
git merge feature-login

Deleting a Branch

After merging, you can delete the feature branch:

Terminal 
git branch -d feature-login
Common Mistake

You can't delete the branch you're currently on. Switch to a different branch first before deleting.

Lessons / Merge vs Rebase

Merge vs Rebase

10 min read

Two Ways to Combine Branches

When you want to bring changes from one branch into another, Git offers two main approaches: merge and rebase. Both accomplish the same goal but work differently.

Git Merge

Merge combines two branches by creating a new "merge commit" that ties them together. It preserves the complete history of both branches.

Terminal 
git checkout main
git merge feature-branch

Merge Visualization

Before
A---B---C feature / D---E---F---G main
After merge
A---B---C feature / \ D---E---F---G---H main (merge commit)

Pros of Merge

  • Preserves complete history
  • Non-destructive operation
  • Shows when branches were combined

Cons of Merge

  • Can create a cluttered history with many merge commits
  • History can be harder to read

Git Rebase

Rebase moves your branch's commits to start from the tip of another branch, creating a linear history as if you started your work later.

Terminal 
git checkout feature-branch
git rebase main

Rebase Visualization

Before
A---B---C feature / D---E---F---G main
After rebase
D---E---F---G main \ A'--B'--C' feature (rebased)

Pros of Rebase

  • Creates a clean, linear history
  • Easier to read and understand
  • No extra merge commits

Cons of Rebase

  • Rewrites commit history
  • Can cause problems if used on shared branches
The Golden Rule of Rebase

Never rebase commits that have been pushed to a public repository. Rebase rewrites history, which can cause serious problems for anyone who has based their work on those commits.

When to Use Which?

Use Merge When:

  • Working on a shared/public branch
  • You want to preserve the complete history
  • Combining long-running branches

Use Rebase When:

  • Working on a local/private branch
  • You want a clean, linear history
  • Keeping your feature branch up to date with main

Lessons / GitHub Basics

GitHub Basics

10 min read

What is GitHub?

GitHub is a web platform that hosts Git repositories online. It adds collaboration features like pull requests, issues, and project management on top of Git.

While Git works locally on your computer, GitHub lets you:

  • Back up your code in the cloud
  • Share projects with others
  • Collaborate with teammates
  • Contribute to open-source projects

Connecting to GitHub

1. Create a Repository on GitHub

  1. Go to github.com and sign in
  2. Click the "+" icon → "New repository"
  3. Name your repository
  4. Choose public or private
  5. Click "Create repository"

2. Connect Your Local Repo

Add GitHub as a remote:

Terminal 
git remote add origin https://github.com/username/repo-name.git

3. Push Your Code

Terminal 
git push -u origin main

The -u flag sets up tracking so future pushes just need git push.

Cloning a Repository

To download an existing repository:

Terminal 
git clone https://github.com/username/repo-name.git

Pulling Changes

Get the latest changes from GitHub:

Terminal 
git pull origin main

The GitHub Workflow

  1. Fork - Copy someone else's repository to your account
  2. Clone - Download the repo to your computer
  3. Branch - Create a branch for your changes
  4. Commit - Make and save your changes
  5. Push - Upload your branch to GitHub
  6. Pull Request - Ask the owner to review and merge your changes
